Автономный оркестратор кодирующих агентов, работающий поверх [OpenCode](https://opencode.ai) и [Beads](https://github.com/steveyegge/beads). Задайте цель, уходите заниматься своими делами, а возвращайтесь к готовой ветке с пройденными тестами.

<br />

## Зачем vtinker?

- **Рекурсивная декомпозиция задач** — планирование, уточнение, выполнение, ревью, исправление, повтор
- **Независимость от модели** — работает с любой моделью, которую поддерживает OpenCode (GLM, Qwen, MiniMax, Kimi, ...)
- **Состояние хранится в Beads**, а не в памяти — каждая задача, вердикт и попытка исправления отслеживается как issue
- **Обнаружение зацикливания** — распознаёт, когда агент застрял, повторяя одно и то же неудачное исправление, и прекращает сжигать токены
- **Продолжение с любой точки** — прервались в 2 часа ночи? `vtinker resume` подхватит ровно с того места, где остановились
- **Потоковый вывод в реальном времени** — наблюдайте за вызовами инструментов и ответами модели по мере выполнения каждой фазы
- **Маршрутизация моделей по фазам** — используйте мощную модель для планирования, быструю для выполнения, тщательную для ревью

## Быстрый старт

```bash
# 1. Установите необходимые компоненты
#    - OpenCode CLI: https://opencode.ai
#    - Beads CLI:    https://github.com/steveyegge/beads
#    - Python 3.11+

# 2. Установите vtinker
pip install -e .

# 3. Перейдите в ваш проект
cd /path/to/your/project

# 4. Создайте конфигурацию (необязательно — vtinker автоматически определит большинство параметров)
mkdir -p .vtinker
cat > .vtinker/config.json << 'EOF'
{
  "checks": [
    {"name": "build", "command": "go build ./..."},
    {"name": "test",  "command": "go test ./..."}
  ]
}
EOF

# 5. Запустите
vtinker start
```

Или пропустите интерактивный диалог полностью:

```bash
vtinker start --title "Add rate limiting to API" --desc "Implement token bucket rate limiter for all /api/* endpoints"
```

Или передайте файл со спецификацией:

```bash
vtinker start --from epic.md
```

## Как это работает

```
 DIALOG ── PREPARE ── PLAN ──┐
                              │
                    ┌─────────┘
                    │
                    ├── REFINE ── EXECUTE ── REVIEW ── FIX ──┐
                    │                                         │
                    └─────────────────────────────────────────┘
                              │
                        FINAL REVIEW
```

| Фаза | Что происходит |
|------|---------------|
| **DIALOG** | Модель исследует кодовую базу, формулирует эпик с критериями приёмки и командами проверки |
| **PREPARE** | Создаёт git-ветку (или worktree) для изолированной работы |
| **PLAN** | Разбивает эпик на упорядоченные задачи с учётом зависимостей и критериев приёмки |
| **REFINE** | Оценивает каждую задачу — атомарные выполняются напрямую, сложные разбиваются на подзадачи |
| **EXECUTE** | Модель реализует одну задачу, руководствуясь описанием, критериями приёмки и контекстом выполненной работы |
| **REVIEW** | Модель проверяет git diff на соответствие критериям приёмки; проверки (сборка/тесты/линтер) запускаются автоматически |
| **FIX** | Если ревью не пройдено, модель получает обратную связь и diff, после чего исправляет проблемы |
| **FINAL REVIEW** | Полный diff проверяется на соответствие исходному эпику; любые пробелы порождают новые задачи, и цикл продолжается |

Каждая фаза выводит данные в stderr, чтобы вы могли наблюдать за прогрессом в реальном времени.

## Конфигурация

Разместите файл конфигурации по пути `.vtinker/config.json` (или `vtinker.json` в корне проекта):

```jsonc
{
  // Рабочая директория (по умолчанию: текущая директория)
  "workdir": ".",

  // Префикс git-ветки для рабочей ветки
  "branch_prefix": "vtinker/",

  // Использовать git worktree вместо переключения веток
  "use_worktree": false,

  // Максимальное количество попыток исправления на задачу
  "max_retries": 10,

  // Тайм-аут на один вызов OpenCode в секундах
  "opencode_timeout": 900,

  // Директория с пользовательскими шаблонами промптов
  "prompts_dir": null,

  // Команды, которые должны проходить после каждой задачи
  "checks": [
    {"name": "build", "command": "go build ./..."},
    {"name": "test",  "command": "go test ./..."},
    {"name": "lint",  "command": "golangci-lint run"}
  ],

  // Модель по умолчанию для всех фаз
  "opencode": {
    "model": "glm-5",
    "agent": null
  },

  // Переопределение модели по фазам (необязательно)
  // Фазы без указанной модели используют opencode.model
  "models": {
    "plan":    "glm-5",        // используется для PLAN + REFINE
    "execute": "glm-4.7",     // используется для EXECUTE + FIX
    "review":  "glm-5"        // используется для REVIEW + FINAL REVIEW
  }
}
```

## Результаты тестирования

Модели протестированы на реальных задачах по написанию кода под управлением vtinker:

| Модель | Время | Выполнено задач | Написано тестов | Попыток исправления | Качество |
|:-------|------:|:---------------:|:---------------:|:-------------------:|:--------:|
| GLM-5 | 30м | 8/8 | 51 | 0 | **A+** |
| GLM-4.7 | 25м | 7/7 | 74 | 1 | **A** |
| MiniMax-m2.7 | 41м | 8/8 | 22 | 1 | **B-** |
| MiniMax-m2.5 | 45м | 5/5 | 19 | 3 | **C** |
| Kimi K2.5 | DNF | - | - | - | **F** |
| Qwen3-coder | 9м | 1/1 | 3 | 1 | **D** |

> GLM-5 выполнил все задачи с первой попытки, без единого исправления. Kimi K2.5 не завершил работу (зацикливание).

## Архитектура

```
vtinker/
  cli.py          Точка входа CLI — команды start, resume, status
  orchestrator.py Основной цикл: фазы, конечный автомат, интеграция с git
  config.py       Загрузка конфигурации (.vtinker/config.json) + сохранение состояния
  beads.py        Тонкая обёртка над CLI Beads (bd)
  opencode.py     Управление процессом OpenCode с потоковым выводом JSONL
  prompts.py      Шаблоны промптов для каждой фазы (переопределяемые)
  parse.py        Парсер структурированного вывода — блоки кода, вердикты, секции
  checks.py       Запуск сконфигурированных проверочных команд, форматирование результатов
  doom.py         Детектор зацикливания (обнаружение повторяющихся ошибок по хешам)
  gitignore.py    Автоуправление .gitignore для артефактов vtinker
```

## Пользовательские промпты

Переопределите промпт любой фазы, разместив Markdown-файл в директории промптов:

```bash
mkdir -p .vtinker/prompts
```

```
.vtinker/prompts/
  dialog.md        # Мастер формулировки задачи
  plan.md          # Декомпозиция эпика на задачи
  refine.md        # Решение: атомарная задача или разбиение
  execute.md       # Инструкции по реализации
  review.md        # Критерии ревью кода
  fix.md           # Инструкции по исправлению на основе обратной связи
  final_review.md  # Полное ревью эпика
```

Каждый файл должен содержать те же слоты `{placeholder}`, что и шаблоны по умолчанию. Полный список слотов для каждой фазы смотрите в [`vtinker/prompts.py`](vtinker/prompts.py).

Укажите путь к директории в конфигурации:

```json
{
  "prompts_dir": ".vtinker/prompts"
}
```

## Справочник по CLI

```
vtinker start [--config PATH] [--dir PATH] [--title TEXT] [--desc TEXT] [--from FILE]
vtinker resume [EPIC_ID] [--config PATH] [--dir PATH]
vtinker status [EPIC_ID]
```

| Команда | Описание |
|---------|----------|
| `start` | Начать новый сеанс vtinker (интерактивный или безголовый) |
| `resume` | Продолжить прерванный сеанс с сохранённого состояния |
| `status` | Показать прогресс эпика и выполнение задач |
